# TamperShield (防篡改之盾) - AI 研发约束指南

## 1. 项目定位与核心原则
* **项目目标**：构建“一对多”跨模态工程文档（扫描件总文件 vs 原生电子件子文件）的防篡改比对流水线。主要处理极其复杂的工程造价清单。
* **绝对真实性原则**：工程审计容不得半点幻觉。**严禁**在数据提取和逻辑比对环节引入任何大语言模型（LLM）进行“语义猜测”或“表格脑补”。必须严格基于视觉坐标、图像处理和物理规则提取。

## 2. 强制技术栈架构
任何新增代码必须遵循以下库的使用规范，不得擅自引入功能重叠的第三方包：
* **图像预处理**：`opencv-python`。
  * *规则*：必须使用 HSV 掩码处理红黑分离（去红章）；必须使用霍夫变换或最小外接矩形处理倾斜校正（Deskew）。
* **表格与结构提取**：`paddlepaddle` + `paddleocr`。
  * *规则*：必须调用 PP-Structure，并利用其输出的 HTML 结构来精准识别跨行/跨列的合并单元格。
* **原生文档解析**：`PyMuPDF` (提取文本) 和 `pdfplumber` (无损提取原生表格)。
* **数据对齐与比对**：`pandas` + `python-Levenshtein`。
  * *规则*：所有提取的数据进入比对前，**必须**统一转化为 `pandas.DataFrame`。面对合并单元格，必须使用 `df.ffill()` 展开以对齐主键。文本容错必须基于 Levenshtein 编辑距离计算。

## 3. 代码规范与目录约束
* **统一数据流**：离散的解析结果在进入 `align_compare.py` 前必须完成 DataFrame 化。
* **模块化**：严禁将所有逻辑塞入 `main.py`。`main.py` 仅作为 Pipeline 的宏观调度中心。
* **目录守则**：
  * 测试输入放入 `data/raw_scans/` 和 `data/base_docs/`
  * 导出报告和中间清洗图片一律输出到 `data/output/`

## 4. AI 助手行为准则
* 在修改或新增代码前，必须先阅读本文件以及 `core/` 目录下的现有代码逻辑。
* 遇到跨行/跨列表格的处理请求时，首选 Pandas 逻辑对齐，切勿试图在物理图像层级进行强行像素切割。
* 你是助手，不是决策者。不要替用户决定"用哪个算法"或"选哪个库"，除非用户明确指定。在提供方案时，务必列出所有已安装的库供选择。
* **绝对禁止在未经明确许可的情况下执行任何写入、修改或覆盖操作。** 仅允许读取文件。任何写入操作（如 save, write, export, overwrite）必须先获得用户明确的确认。如果无法确认，请输出 "[Output Blocked] - Permission Required" 并停止操作。
* **不要主动创建或覆盖文件。** 你唯一的任务是执行分析或提供信息。如果你发现需要一个中间文件来完成任务，必须询问用户是否需要创建该文件。
* **避免默认行为。** 当面对需要保存结果的任务时，不要自动生成文件名或路径。必须要求用户明确指定保存位置和文件名，然后将路径返回给用户进行确认。
* **保持简洁。** 除非被要求，否则不要添加解释性文字或额外的上下文。直接输出分析结果或代码。
